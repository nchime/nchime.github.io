# Skill: 블로그 포스트 생성기 (Blog Post Generator)

이 스킬은 워크스페이스 내의 블로그 스타일(문체, 프론트매터 구성, 문서 구조, 컴포넌트 활용 등)을 분석하여, 기존 블로그와 일관된 품질과 형식을 갖춘 MDX 블로그 포스트(`data/blog/YYYYMMDD-slug.mdx`)를 자동 생성하는 AI 에이전트 전용 스킬입니다.

---

## 1. 사전 요구사항 및 입력 포맷

에이전트는 블로그 글을 생성하기 위해 다음과 같은 정보를 입력받아야 합니다:
1. 주제 및 핵심 키워드: 예) "AI 시대의 온톨로지(Ontology) 개념 정리", "하네스 엔지니어링 소개"
2. 대상 독자: 예) 풀스택 개발자, AI 엔지니어, 기술 관심층 등
3. 참조 자료 또는 소스 코드: 블로그 글에 녹여낼 기술적 상세 사양이나 참조 링크

---

## 2. 파일 명명 및 경로 규칙

- 저장 경로: `data/blog/`
- 파일명 형식: `YYYYMMDD-topic-slug.mdx`
  - 날짜는 현재 날짜 기준 `YYYYMMDD` 형식을 따릅니다. (예: `20260523-ontology-service.mdx`)
  - 슬러그(Slug)는 영어 소문자, 숫자, 하이픈(-)만 사용하여 생성합니다.
  - 하루에 여러 포스트를 작성하는 경우, 날짜 뒤에 순번을 붙여 구별합니다. (예: `2025021501-nextjsmdx.mdx`)

---

## 3. 프론트매터(Frontmatter) 스키마 규칙

모든 블로그 포스트는 파일 최상단에 다음과 같은 형식의 YAML 프론트매터를 포함해야 합니다.

```yaml
---
title: '[블로그 글 제목: 핵심 내용을 직관적으로 표현하는 한글 제목]'
date: 'YYYY-MM-DD'
tags: ['기술1', '기술2', '키워드']
images: ['대표 이미지 경로 또는 Unsplash URL']
draft: false
summary: '블로그 포스트의 내용을 1~2문장으로 요약한 설명입니다.'
---
```

- title: 따옴표`'`로 감싸며, 제목과 부제 형식을 권장합니다. (예: `'하네스 엔지니어링(Harness Engineering): AI 에이전트 팀 구축을 위한 혁신적 프레임워크'`)
- date: 싱글 쿼트`'`로 감싼 `YYYY-MM-DD` 형식이어야 합니다.
- tags: 대소문자를 구분하는 기술 키워드 문자열 배열입니다. (예: `['AI', 'Ontology', 'RAG']`)
- images: 포스트 카드나 공유 시 사용될 대표 이미지의 경로입니다.
  - 로컬 이미지인 경우 `/static/images/` 디렉토리를 참조합니다.
  - 외부 이미지인 경우 고품질 Unsplash 등의 URL을 지정합니다.
- draft: 배포용인 경우 `false`로 설정합니다.
- summary: 블로그 홈과 SEO 메타 설명에 사용되므로 정중하고 명확한 문체로 요약합니다.

---

## 4. 본문 내용 및 스타일 가이드

에이전트는 작성 시 기존 블로그 글들의 문체와 포맷팅 특징을 그대로 계승해야 합니다.

### 문체 및 어조 (Tone of Voice)
- 존칭 사용: 독자에게 지식을 공유하는 친절하고 전문적인 어조를 유지합니다 (`~합니다`, `~입니다`, `~것입니다`).
- 명확성: 모호한 설명은 피하고, 핵심 용어를 명확히 전달합니다.
- 시각적 강조: 중요한 결론이나 원칙은 인용구(`>`)나 표(Table), ASCII/텍스트 다이어그램을 활용하여 쉽게 읽히도록 합니다.
- 강조 마커 금지: 본문 내에서 볼드체 강조를 표현하기 위한 더블 아스테리스크(예: `**텍스트**`) 마크다운 문법은 사용하지 않습니다. 텍스트를 강조할 때는 문맥 상 자연스럽게 기술하거나 코드 블록, 인용구를 대신 활용합니다.

### 본문 구조 (Document Structure)
1. 도입부 (Introduction): 독자의 관심을 끌고 글의 목적과 전체 목차/요약을 1~2문단으로 정중하게 제시합니다.
2. 구분선: 도입부와 첫 번째 대주제 사이에는 가로 구분선(`---`)을 삽입합니다.
3. 헤더 규칙: 
   - 대주제는 `## 1. 주제명`, `## 2. 주제명` 순서로 번호를 매깁니다.
   - 소주제는 `### 3-1. 소주제명`, `### • 강조 사항` 등으로 작성합니다.
   - 각 대주제 섹션 끝에는 가로 구분선(`---`)을 배치하여 가독성을 높입니다.
4. 결론부 (Conclusion): `## 마치며` 또는 `## 결론` 섹션을 구성하여 전체 내용을 요약하고 앞으로의 방향성을 제안합니다.
5. 참고 링크: 포스트 마지막에 `참고 링크:`라는 문구 밑에 관련 표준 스펙, 공식 문서, 라이브러리 저장소 링크를 마크다운 링크 형식으로 명시합니다. 이때 `참고 링크:` 텍스트 자체에도 더블 아스테리스크를 사용하지 않습니다.

### 컴포넌트 및 마크다운 확장 활용
본 프로젝트는 MDX 환경으로, 본문 작성 시 풍부한 구조를 제공하기 위해 다음 요소를 적극 활용합니다.
1. 비교 표(Markdown Table): 
   개념 비교 시 표 형식을 사용하여 직관적으로 표현합니다. 이때 표 머리글이나 내용에도 `**`를 사용하지 않습니다.
   예시:
   | 구분 | 개념 A | 개념 B |
   | :--- | :--- | :--- |
   | 강점 | 설명 | 설명 |
2. 텍스트 기반 다이어그램/구조도:
   프로세스나 구조를 한눈에 볼 수 있도록 텍스트 블록으로 다이어그램을 표현합니다.
   예시:
   ```
   [입력 데이터] → (처리 엔진) → [구조화된 결과]
   ```
3. 내장 컴포넌트 활용:
   인터랙티브하거나 동적인 요소가 필요한 경우 다음 리액트 컴포넌트를 직접 사용하거나 가져올 수 있습니다:
   - `<InteractiveBox />`
   - `<AgGridComponent />`
   - `<EChartComponent />`
   - `<StyledButton images={['이미지URL1', '이미지URL2']} />` (상단에 `import StyledButton from '@/components/StyledButton'` 추가 필요)

## 5. 생성 검증 체크리스트

포스트 생성이 완료된 후 에이전트는 다음 사항을 최종 검증해야 합니다.
- [ ] 파일 이름이 `YYYYMMDD-slug.mdx` 형식으로 올바른 경로(`data/blog/`)에 생성되었는가?
- [ ] 프론트매터의 `title`, `date`, `tags`, `images`, `draft`, `summary`가 모두 채워져 있고 형식에 맞는가?
- [ ] 제목과 소제목이 정해진 번호 양식(`## 1.`, `## 2.`, `### 3-1.`)을 따르고 있는가?
- [ ] 한국어 문법 및 맞춤법이 올바르며, 전문적인 개발 블로그 문체를 유지하고 있는가?
- [ ] 본문 전체에 볼드 강조용 더블 아스테리스크(`**`) 표시가 단 하나도 사용되지 않았는가?
- [ ] 하단에 `## 마치며`와 `참고 링크:`가 빠짐없이 작성되었는가?
