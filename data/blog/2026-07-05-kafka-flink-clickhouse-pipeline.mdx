## 1. 왜 이 조합인가?

현대 데이터 플랫폼에서 **실시간**은 더 이상 사치가 아니라 필수입니다. 사용자 행동 분석, IoT 센서 모니터링, 실시간 대시보드, 사기 탐지 — 어떤 유스케이스든 이벤트가 발생하는 순간 분석 결과를 도출해야 합니다.

이 문제를 풀기 위해 널리 쓰이는 조합이 있습니다:

| 레이어 | 역할 | 대표 기술 |
|:-------|:-----|:----------|
| **Ingestion** | 대량 이벤트 수집 및 버퍼링 | Apache Kafka |
| **Processing** | 상태 기반 스트림 처리 | Apache Flink |
| **Storage & Query** | 컬럼형 OLAP 고속 분석 | ClickHouse |

각 컴포넌트의 강점을 극대화하면서 약점을 상호 보완하는 구조입니다.

---

## 2. 전체 아키텍처 개요

![Kafka → Flink → ClickHouse 파이프라인 아키텍처](/static/images/kafka-flink-clickhouse/pipeline-architecture.svg)

위 다이어그램에서 보듯, 파이프라인은 크게 4단계로 구성됩니다:

1. **Producer (데이터 소스)**: 웹 앱, 모바일 SDK, IoT 센서 등에서 이벤트 생성
2. **Kafka**: 파티션 기반 병렬 처리로 이벤트를 안정적으로 수집·보유
3. **Flink**: 윈도우 기반 집계, 상태 관리, Exactly-Once 보장
4. **ClickHouse**: 컬럼 기반 저장으로 고속 OLAP 쿼리 수행
5. **시각화/API**: Grafana 대시보드 또는 분석 API로 데이터 제공

## 3. 각 컴포넌트 상세 분석

### 3.1 Apache Kafka — 이벤트 수집 계층

Kafka는 분산 이벤트 스트리밍 플랫폼으로, 이 파이프라인의 **입구** 역할을 합니다.

**핵심 특징:**

- **파티셔닝(Partitioning)**: 토픽을 여러 파티션으로 분산하여 병렬 쓰기/읽기 가능
- **내구성(Durability)**: 디스크에 메시지를 지속적으로 보존하여 재처리 가능
- **스케일아웃**: 브로커 추가만으로 처리량 확장

**실전 설계 예시:**

```java
// Producer 설정
Properties props = new Properties();
props.put("bootstrap.servers", "kafka-1:9092,kafka-2:9092,kafka-3:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("acks", "all");  // 내구성 보장
props.put("compression.type", "lz4");  // 전송 효율

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

// 이벤트 발행
ProducerRecord<String, String> record = new ProducerRecord<>(
    "user-events",      // 토픽
    userId,             // 파티셔닝 키
    eventJson           // 이벤트 페이로드
);
producer.send(record);
```

**토픽 설계 전략:**

| 토픽명 | 파티션 수 | 정책 | 용도 |
|:-------|:----------|:-----|:-----|
| `user-events` | 12 | 7일 보존 | 사용자 클릭/페이지뷰 |
| `order-events` | 6 | 30일 보존 | 주문/결제 이벤트 |
| `sensor-telemetry` | 24 | 3일 보존 | IoT 센서 데이터 |

> **Tip**: 파티션 수는 Flink 병렬도와 동일하게 설정하면 컨슈머 리밸런싱 비용을 줄일 수 있습니다.

### 3.2 Apache Flink — 스트림 처리 엔진

Flink는 이 파이프라인의 **핵심 처리 계층**입니다. 단순한 ETL을 넘어, 상태 기반 처리와 Exactly-Once 시맨틱을 제공합니다.

**왜 Flink인가?**

| 기능 | Flink | Spark Streaming | Kafka Streams |
|:-----|:------|:----------------|:--------------|
| 처리 모델 | True Streaming (Event-at-a-time) | Micro-batch | Streaming |
| 상태 관리 | 내장 (RocksDB 백엔드) | 외부 필요 | 내장 (제한적) |
| 장애 복구 | Checkpoint + Savepoint | WAL | 로그 기반 |
| Exactly-Once | End-to-end 보장 | At-least-once (기본) | 내부만 보장 |
| 윈도우 | 이벤트 시간 기반 풍부한 윈도우 | 제한적 | 기본 제공 |

**Flink Job 예시 — 실시간 사용자 세션 집계:**

```java
public class UserSessionAggregation {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // 체크포인트 설정 ( Exactly-Once 보장 )
        env.enableCheckpointing(60_000);  // 1분 간격
        env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
        env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000);
        env.getCheckpointConfig().setCheckpointStorage("s3://flink-checkpoints/");

        // Kafka 소스
        KafkaSource<String> source = KafkaSource.<String>builder()
            .setBootstrapServers("kafka-1:9092,kafka-2:9092")
            .setTopics("user-events")
            .setGroupId("flink-session-agg")
            .setStartingOffsets(OffsetsInitializer.committedOffsets(OffsetResetStrategy.LATEST))
            .setValueOnlyDeserializer(new SimpleStringSchema())
            .build();

        DataStream<UserEvent> events = env.fromSource(
            source,
            WatermarkStrategy.<String>forBoundedOutOfOrderness(Duration.ofSeconds(10))
                .withTimestampAssigner((event, ts) -> extractTimestamp(event)),
            "Kafka Source"
        ).map(json -> objectMapper.readValue(json, UserEvent.class));

        // 이벤트 시간 기반 세션 윈도우 (30분 비활성 시 세션 종료)
        DataStream<SessionMetric> sessionMetrics = events
            .keyBy(UserEvent::getUserId)
            .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
            .aggregate(new SessionAggregator(), new SessionWindowFunction());

        // ClickHouse Sink
        sessionMetrics.addSink(new ClickHouseSink());

        env.execute("User Session Aggregation");
    }
}
```

**Flink의 핵심 개념 정리:**

```
이벤트 타임(Event Time) vs 처리 타임(Processing Time)
├── 이벤트 타임: 이벤트가 실제로 발생한 시간 (정확도 높음)
└── 처리 타임: 시스템에서 처리된 시간 (.SimpleButton 낮은 지연시간)

윈도우 유형
├── Tumbling Window: 고정 크기, 겹침 없음 (예: 1분마다 집계)
├── Sliding Window: 고정 크기, 슬라이딩 간격으로 겹침
├── Session Window: 비활성 간격(Gap) 기반 동적 크기
└── Global Window: 모든 요소를 하나의 윈도우로 (Custom Trigger 필요)

상태 관리
├── ValueState: 단일 값 상태
├── ListState: 리스트 상태
├── ReducingState: 리듀스 연산 상태
├── AggregatingState: 집계 상태
└── MapState: Map 형태 상태
```

### 3.3 ClickHouse — OLAP 분석 엔진

ClickHouse는 이 파이프라인의 **분석 최종 목적지**입니다. 컬럼형 저장소 기반으로, 대량 데이터에서 초고속 집계 쿼리를 수행합니다.

**왜 ClickHouse인가?**

- **컬럼형 저장**: 필요한 컬럼만 읽으므로 집계 쿼리가 극도로 빠름
- **압축률**: 행 기반 DB 대비 10배 이상 높은 압축률로 스토리지 절감
- **병렬 처리**: 쿼리 한 건이 자동으로 여러 코어에서 병렬 실행
- **실시간 적재**: `Buffer` 엔진을 통한 실시간 데이터 적재 지원

**테이블 설계:**

```sql
-- 이벤트 원본 테이블 (MergeTree 엔진)
CREATE TABLE user_events_local ON CLUSTER '{cluster}'
(
    event_id        String,
    user_id         UInt64,
    event_type      LowCardinality(String),
    page_url        String,
    event_time      DateTime64(3),
    event_date      Date DEFAULT toDate(event_time),
    properties      Map(String, String),
    device_type     LowCardinality(String),
    country_code    LowCardinality(FixedString(2)),
    session_id      String
)
ENGINE = ReplicatedMergeTree('/clickhouse/{cluster}/tables/{shard}/user_events', '{replica}')
PARTITION BY toYYYYMM(event_date)
ORDER BY (user_id, event_type, event_time)
TTL event_date + INTERVAL 90 DAY
SETTINGS index_granularity = 8192;

-- 분산 테이블
CREATE TABLE user_events ON CLUSTER '{cluster}'
AS user_events_local
ENGINE = Distributed('{cluster}', 'default', 'user_events_local', xxHash64(user_id));

-- 실시간 버퍼 테이블 (Flink → ClickHouse 적재용)
CREATE TABLE user_events_buffer ON CLUSTER '{cluster}'
AS user_events_local
ENGINE = Buffer('default', 'user_events_local', 16, 10, 100, 10000, 100000, 1000000, 10000000);
```

**고성능 쿼리 예시:**

```sql
-- 실시간 사용자 활동 대시보드 (최근 1시간)
SELECT
    toStartOfInterval(event_time, INTERVAL 5 MINUTE) AS time_bucket,
    event_type,
    uniqExact(user_id) AS unique_users,
    count() AS total_events
FROM user_events
WHERE event_time >= now() - INTERVAL 1 HOUR
GROUP BY time_bucket, event_type
ORDER BY time_bucket DESC, total_events DESC;

-- 사용자 세션 분석 (평균 세션 길이, 이벤트 수)
SELECT
    user_id,
    min(event_time) AS session_start,
    max(event_time) AS session_end,
    dateDiff('second', session_start, session_end) AS session_duration_sec,
    count() AS events_per_session
FROM user_events
WHERE event_date = today()
GROUP BY user_id
ORDER BY session_duration_sec DESC
LIMIT 100;

-- 장비별/국가별 실시간 활성 사용자
SELECT
    device_type,
    country_code,
    uniqExact(user_id) AS dau
FROM user_events
WHERE event_time >= now() - INTERVAL 24 HOUR
GROUP BY device_type, country_code
ORDER BY dau DESC;
```

## 4. End-to-End Exactly-Once 보장

이 파이프라인에서 가장 중요한 비기능적 요구사항 중 하나는 **End-to-End Exactly-Once Semantics**입니다. 같은 이벤트가 중복 처리되거나 유실되면 안 됩니다.

### 4.1 Kafka → Flink 구간

```
Flink Checkpoint Mechanism:
┌─────────────────────────────────────────────────┐
│  1. Barrier 삽입: Flink가 Kafka offset에        │
│     Barrier를 삽입하여 스트림을 분할             │
│                                                  │
│  2. 상태 스냅샷: 모든 Operator의 상태를          │
│     안정 스토리지(S3/HDFS)에 저장               │
│                                                  │
│  3. Offset 커밋: Checkpoint 완료 후              │
│     Kafka offset를 커밋                         │
│                                                  │
│  4. 장애 시: Checkpoint에서 복원하여              │
│     Kafka offset를 재설정 → 재처리              │
└─────────────────────────────────────────────────┘
```

### 4.2 Flink → ClickHouse 구간

```java
// Two-Phase Commit Pattern으로 ClickHouse 적재
public class ExactlyOnceClickHouseSink
    extends TwoPhaseCommitSinkFunction<Event, Void, Void> {

    @Override
    protected Void beginTransaction() throws Exception {
        // ClickHouse에 새 배치 시작
        return clickhouseClient.beginBatch();
    }

    @Override
    protected void invoke(Void transaction, Event event, Context context) {
        // 이벤트를 배치에 추가
        clickhouseClient.addToBatch(transaction, event);
    }

    @Override
    protected void preCommit(Void transaction) {
        // 배치 데이터 플러시
        clickhouseClient.flushBatch(transaction);
    }

    @Override
    protected void commit(Void transaction) {
        // ClickHouse에 커밋 확인
        clickhouseClient.commitBatch(transaction);
    }

    @Override
    protected void abort(Void transaction) {
        // 실패 시 배치 롤백
        clickhouseClient.rollbackBatch(transaction);
    }
}
```

## 5. 성능 튜닝 가이드

### 5.1 Kafka 튜닝

| 파라미터 | 권장값 | 설명 |
|:---------|:-------|:-----|
| `num.partitions` | Flink 병렬도와 동일 | 리밸런싱 최소화 |
| `replication.factor` | 3 | 내구성 보장 |
| `batch.size` | 65536 ~ 131072 | 배치 전송으로_throughput_ 향상 |
| `linger.ms` | 5 ~ 20 | 배치 대기 시간 |
| `compression.type` | lz4 | CPU/대역폭 트레이드오프 |

### 5.2 Flink 튜닝

| 파라미터 | 권장값 | 설명 |
|:---------|:-------|:-----|
| `parallelism` | 파티션 수와 일치 | 리밸런싱 최소화 |
| `state.backend` | rocksdb | 대용량 상태 처리 |
| `state.checkpoints.dir` | s3://bucket/path | 분산 스토리지에 체크포인트 |
| `checkpointing.interval` | 60000ms | 정확도/오버헤드 트레이드오프 |
| `buffer.timeout` | 10ms | 지연시간 최소화 |

### 5.3 ClickHouse 튜닝

```sql
-- 1. 파티션 전략: 날짜 기반 분할로 쿼리 스캔 범위 최소화
PARTITION BY toYYYYMM(event_date)

-- 2. 정렬 키: 쿼리 패턴에 맞게 설계
ORDER BY (user_id, event_type, event_time)

-- 3. 보조 인덱스 (Secondary Index)
ALTER TABLE user_events ADD INDEX idx_event_type event_type TYPE set(100) GRANULARITY 4;

-- 4. Skipping Index: 불필요한Granule 건너뛰기
ALTER TABLE user_events ADD INDEX idx_country country_code TYPE set(10) GRANULARITY 4;

-- 5. 메모리 설정
SET max_threads = 16;
SET max_memory_usage = 10000000000;
SET max_bytes_before_external_group_by = 5000000000;
```

## 7. 장애 대응 전략

### 7.1 각 컴포넌트 장애 시나리오

| 장애 유형 | 영향 | 복구 방법 |
|:----------|:-----|:----------|
| Kafka 브로커 장애 | 파티션 리더 재선거 | Replication Factor 3으로 자동 복구 |
| Flink TaskManager 장애 | 체크포인트에서 자동 복원 | Checkpoint 간격에 따라 RTO 결정 |
| ClickHouse 노드 장애 | ReplicatedMergeTree 자동 페일오버 | ZKeeper 기반 복제 자동 복구 |
| 전체 장애 (Disaster) | Savepoint에서 전체 재시작 | `flink run -s s3://savepoints/...` |

### 7.2 백업 전략

```bash
# Flink Savepoint 생성 (계획적 재시작용)
bin/flink savepoint <job-id> s3://flink-savepoints/

# ClickHouse 백업 (clickhouse-backup 도구)
clickhouse-backup create daily_backup_$(date +%Y%m%d)
clickhouse-backup upload daily_backup_$(date +%Y%m%d)
```

## 9. 대안과 비교

| 조합 | 장점 | 단점 |
|:-----|:-----|:-----|
| **Kafka + Flink + ClickHouse** | 초실시간 + 고성능 분석 + Exactly-Once | 운영 복잡도 높음 |
| Kafka + Spark Streaming + Druid | Spark 생태계 활용 가능 | Micro-batch 지연시간, Druid 운영 비용 |
| Kafka + Flink + Elasticsearch | 텍스트 검색에 강함 | 대규모 집계 성능 제한, 스토리지 비용 높음 |
| Kafka Streams + ClickHouse | 아키텍처 단순 | 상태 관리 및 장애 복구 기능 제한 |
| Confluent + Flink + ClickHouse | 상용 지원 + 관리형 서비스 | 라이선스 비용 |

## 10. 요약

Kafka + Flink + ClickHouse 조합은 **대규모 이벤트 데이터의 실시간 수집, 처리, 분석**을 위한 검증된 아키텍처입니다.

**핵심 설계 원칙:**

1. **관심사 분리**: 각 컴포넌트가 하나의 역할에 최적화
2. **데이터 버퍼링**: Kafka가 피크 트래픽을 흡수하여 안정성 확보
3. **정확한 처리**: Flink의 Exactly-Once로 데이터 무결성 보장
4. **고속 분석**: ClickHouse의 컬럼형存储으로 초고속 집계 쿼리
5. **장애 허용 설계**: 각 레이어의 내재적 복구 메커니즘 활용

이 파이프라인을 운영하면서 가장 중요한 것은 **지속적인 모니터링과 튜닝**입니다. 메트릭을 꾸준히 관찰하고, 쿼리 패턴에 맞게 파티셔닝과 정렬 키를 최적화하며, 체크포인트 간격과 배치 크기를 조정하는 것이 장기적 성공의 열쇠입니다.
